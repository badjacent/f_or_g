# Subway Home Router --- Implementation Principles

The following principles are **mandatory architectural and product
constraints** for the system. They exist to keep the application
reliable, predictable, and extremely simple from the user's perspective.

------------------------------------------------------------------------

## 1. Separate feed parsing from ETA logic

The system must have a clear separation between:

1.  **Transit feed ingestion**
2.  **Transit model normalization**
3.  **Routing / ETA decision logic**

The routing engine must **not depend on raw MTA feed structures**.

### Required layers

**Feed Layer** - Responsible for retrieving MTA GTFS-realtime feeds and
alerts. - Converts raw feed data into internal normalized objects.

**Transit Model Layer** Defines internal types such as: -
`TrainPrediction` - `RoutePrediction` - `StopArrival` - `ServiceAlert`

This layer contains **no routing logic**.

**Decision Engine** - Consumes only normalized transit objects. -
Computes route candidates and selects the best option.

This separation prevents transit feed format changes from breaking
routing logic and keeps the decision algorithm testable.

------------------------------------------------------------------------

## 2. Represent recommendation reasons as enums

Recommendation logic must produce **structured internal states**, not ad
hoc text strings.

Example enum:

    RecommendationReason {
      FASTEST_CLEAR
      FASTEST_TIGHT_TRANSFER
      ABOUT_THE_SAME_PREFER_EASIER
      LOW_CONFIDENCE
      DATA_UNAVAILABLE
    }

The backend must return both:

-   `recommendedRoute`
-   `recommendationReason`

Example:

    {
      recommendedRoute: "F",
      recommendationReason: "FASTEST_TIGHT_TRANSFER"
    }

User-facing copy must be generated by mapping these enums to UI text.\
Routing logic must **never construct UI strings directly**.

------------------------------------------------------------------------

## 3. Do NOT use stale-while-revalidate

The system must **not display stale recommendations** while recomputing.

Reasons:

-   Recommendations may flip if the next train prediction changes.
-   Showing an old recommendation and then immediately changing it
    creates confusion.

Required behavior:

-   When the user opens the app, the backend computes a **fresh
    decision**.
-   The client waits for that response.
-   The UI may show a **loading state**, but must not show cached route
    advice.

Small backend caches of **transit feed data** are acceptable, but the
**recommendation itself must be freshly computed per request**.

------------------------------------------------------------------------

## 4. Backend decides the full recommendation state

The backend must compute the **complete main-screen state**.

The client should remain a thin display layer.

The backend response must contain:

    recommendedRoute        ("F" or "G")
    urgencyState            ("NORMAL" | "HURRY")
    recommendationReason    (enum)
    summaryText             (short explanation)
    etaF
    etaG
    confidenceLevel
    dataFreshnessSeconds
    debugData

The client should **not recompute or reinterpret routing logic**.\
It should simply render the backend response.

------------------------------------------------------------------------

## 5. Build a hidden debug screen early

A hidden debug screen must exist in the client for development and
troubleshooting.

Example access methods:

-   triple tap on the main screen
-   long press on recommendation letter
-   debug build flag

The debug screen should display:

### Decision inputs

-   current timestamp
-   feed timestamp
-   alerts affecting A/F/G
-   platform arrival assumptions

### Candidate analysis

Example:

    route: F
    etaMinutes: 18
    transferMarginSeconds: 63
    tightTransfer: true
    candidateTrainsConsidered: [...]

### Decision result

    winningRoute
    recommendationReason
    urgencyState
    etaDifference
    confidenceScore

This screen is critical for diagnosing bad recommendations.

------------------------------------------------------------------------

## 6. Treat tight transfers as a UX concept

Transfer tightness is **not only a mathematical threshold**. It must
drive both the routing logic and the UI state.

Define:

    transferMargin = trainArrivalTime - riderArrivalToPlatform

Suggested interpretation:

  Margin       Classification
  ------------ ----------------
  \>= 90 sec   Comfortable
  45--90 sec   Tight
  \< 45 sec    Very unlikely

Rules:

-   Tight transfers are **still valid candidates**.
-   If the winning route depends on a tight transfer, the UI must show
    an **urgency state**.

Example UI states:

    F
    F — HURRY
    G
    G — HURRY

Urgency must appear prominently on the main screen.

The goal is **clear action guidance**, not subtle messaging.

------------------------------------------------------------------------

## 7. Protect first paint at all costs

The value of the application depends on **instant clarity**.

Performance goals:

-   Time from app open to recommendation: **\< 1 second on warm
    backend**
-   The first visible screen must show the **recommendation letter
    prominently**.

Strategies:

-   Keep the client extremely lightweight.
-   Perform all heavy logic on the backend.
-   Avoid unnecessary client-side computation.
-   Render the recommendation immediately when the API response arrives.

The interaction should feel like:

    open app
    see giant letter
    act

------------------------------------------------------------------------

## Summary

The architecture must enforce:

-   Clean separation of feed ingestion and routing logic
-   Enum-based reasoning states
-   Fresh recommendations on every request
-   Backend-driven UI state
-   Robust debugging visibility
-   Tight transfers surfaced as explicit urgency
-   Extremely fast first render

The system's single job is to tell the rider whether to take the **F or
the G** as quickly and clearly as possible.

------------------------------------------------------------------------

## Spec Addendum (v2 Decisions)

This addendum is authoritative for implementation details that were
previously ambiguous.

### Product naming and scope

- App name is **F or G**.
- Recommendation targets are only `F`, `G`, or `?` when data is
  unavailable.
- `A` is never a recommendation target.

### Transfer assumptions

- Base assumption: rider reaches `A` platform now.
- `A -> F` transfer overhead: `0` seconds (cross-platform).
- `A -> G` transfer overhead: `90` seconds (buffer included).

### Tie-break policy

- If absolute ETA difference between `F` and `G` is `<= 60` seconds,
  treat as tie.
- On tie, recommend `F`.

### Freshness and fallback

- Feed freshness threshold: `60` seconds.
- If no usable signal/feeds are available, return:
  - `recommendedRoute: "?"`
  - `recommendationReason: "DATA_UNAVAILABLE"`
  - summary text indicating no signal.

### ETA and urgency policy

- ETA strategy is **conservative** (prefer safer arrival estimate over
  optimistic estimate).
- `urgencyState` is determined from transfer margin, not from ETA delta
  alone.

### Confidence policy

Use the following deterministic tiers:

- `HIGH`
  - feed age `<= 60s`
  - both `F` and `G` ETAs available
  - winner not driven by a borderline transfer
  - ETA advantage `> 60s`
- `MEDIUM`
  - feed age `<= 60s`
  - both ETAs available
  - winner depends on transfer-margin sensitivity or close ETA gap
- `LOW`
  - feed age `> 60s` but still usable, or one route ETA is weak/inferred
  - decision depends on fallback/tie handling
- `DATA_UNAVAILABLE`
  - no meaningful ETA computation possible

### Client behavior updates

- Main screen includes a small, visually unobtrusive refresh button near
  bottom.
- Refresh triggers a full fresh backend recompute.
- No stale recommendation is shown while recomputing.

### Debug data scope (minimal required fields)

Include only:

- decision timestamp
- feed timestamp and `dataFreshnessSeconds`
- computed `etaF`, `etaG`
- transfer margins used for winner evaluation
- winning route
- `recommendationReason`
- `urgencyState`
- `confidenceLevel`
